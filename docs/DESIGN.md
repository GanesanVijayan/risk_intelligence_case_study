# DESIGN.md – Risk Intelligence Pipeline

## Stack Overview
- **Language**: Python 3.10+
- **Libraries**:
  - `pdfplumber` → PDF ingestion --> (src/ingestion.py)
  - `spaCy` + regex → candidate detection --> (src/candidate_check.py)
  - LLM API (Azure OpenAI / HuggingFace) → summarization & classification --> (src/summarizer_classifier.py) --> "mistralai/ministral-3-3b" is used
  - `pydantic` → schema validation (`RiskReport`) --> (src/schema.py)
  - `tabulate` → tabular output for user queries --> (src/pipeline.py) --> lightweight model "qwen2.5-0.5b-instruct" is used for intent and entity classification
- **Evaluation**:
  - Custom evaluator pipeline (exact match, semantic similarity, coverage) --> (src/evaluators.py) --> ultra lightweight model "text-embedding-nomic-embed-text-v1.5" used for semantic scoring
  - Golden set JSON for regression detection --> (data/golder_set.json) --> generated by GPT 5.5 (Microsoft Copilot)
- **Logging & Resilience**: --> (src/pipeline.py)
  - `logging` module with file + console handlers
  - Retry decorator for transient failures
  - Fail‑fast schema validation

## Pipeline Decomposition (pipeline.py --> main runner)
1. **Ingestion**
   - Extract sections from PDF using `pdfplumber`.
   - Targeted pages (risk management, material impacts, cyber security, climate change).
2. **Candidate Detection**
   - `spaCy` sentence segmentation + regex for risk keywords.
   - Filters out non‑risk text early.
3. **Summarization & Classification**
   - LLM generates short title, description, category, mitigation.
   - Prompt strength adjustable (`prompt_strenth="stronger"`).
4. **Schema Validation**
   - Pydantic `RiskReport` ensures structured consistency.
   - JSON output persisted per page.
5. **Consolidation**
   - Merge per‑page JSON into consolidated report.
6. **Evaluation**
   - Compare against golden set.
   - Metrics: exact match, semantic similarity, coverage.
   - Thresholds configurable (`exact=50, sem=70, cov=80`).
7. **User Queries**
   - Filter + tabulate results.
   - Example: “filter by category climate from p.72 with mitigation non‑blank”.

## Key Trade‑offs
- **LLM vs Classical NLP**:
  - Chose LLM summarization for richer descriptions and categorization.
  - SpaCy + regex retained for lightweight candidate detection.
- **Schema Enforcement**:
  - Pydantic ensures downstream consistency but adds strictness (fail‑fast).
- **Evaluation Strategy**:
  - Golden set comparison ensures regression detection.
  - Semantic similarity adds robustness but requires embeddings (cost/latency).
- **Sequential vs Parallel Processing**:
  - Current slice: sequential per page for simplicity.
  - Future: parallelization across pages/reports for throughput.

## Scaling Considerations
- **Latency**: Parallelize page processing; batch LLM calls.
- **Cost**: Cache embeddings; reuse classification prompts.
- **Throughput**: Deploy microservices on AKS (Azure) or OpenShift (on‑prem).
- **Observability**: Integrate DeepEval, RAGAs, LangSmith for evaluation metrics.
- **Resilience**: Guardrails (content safety, schema checks), retry/fallback mechanisms.
